Founder report: "the app is currently quite buggy, laggy and unresponsive." This is the full
audit: what is wrong, ranked by user-felt impact, the fix plan, and the guardrails that keep
these classes of problem from coming back.

**Health snapshot at audit time:** `pnpm build`, `pnpm typecheck`, `pnpm lint:content`,
`pnpm test:srs` (323 checks), `pnpm test:pronounce` (26 checks) and `pnpm audit` are ALL green.
So nothing is "broken" in the compiler's eyes; the problems below are architectural and
runtime-behavioral, which is exactly the kind CI currently cannot see (see section D).

## A. Performance: startup weight (why first load and PWA resume feel slow)

### A1. The main bundle is 606 kB minified (174 kB gzip) and carries every content bank
`vite build` output: `index-*.js` = 606 kB. The cause is import topology, not any one file:

- `lib/search.ts` statically imports **all five content banks** (vocabulary 268 kB source,
  collocations 140 kB, dialogues 76 kB, redemittel, grammar) and is itself statically imported
  by `GlobalSearch`, which `AppShell` imports eagerly. So the entire content corpus is on the
  critical path of every page, including `/welcome`.
- `engine/session.ts` (imported by the eagerly-loaded Dashboard) pulls vocabulary, redemittel,
  grammar and the quiz engine.

On a mid-range phone, parsing and evaluating ~600 kB of JS blocks the main thread for hundreds
of ms on top of download. This is the single biggest "app feels slow to open" contributor.

**Fix plan:**
1. Load the search corpus lazily: `GlobalSearch` does `const { searchAll } = await
   import("@/lib/search")` when the dialog first opens (or on `requestIdleCallback` after
   mount). This alone removes dialogues + collocations + redemittel + grammar from the
   critical path for users who never search.
2. Split `dialogues.ts` out of anything eager (it is only needed by SimulationHub and search).
3. Keep vocabulary eager (Dashboard/session legitimately need it) but stop importing whole
   banks where a derived slice would do.
4. Add a CI bundle budget so this never regresses silently (D4).

Expected result: main chunk roughly halves; time-to-interactive on mobile improves accordingly.

## B. Performance: runtime jank (why interacting feels laggy)

### B1. Unvirtualized lists with a framer-motion wrapper per card (highest interaction cost)
`VocabList` renders up to **528** cards, `CollocationsBrowser` up to **396**, each wrapped in
its own `motion.div` with staggered mount animation, plus `SpeakButton`, `Badge`, mastery
computation and `relatedRows(v)` recomputed per render. Two compounding problems:

- **Typing in the browse search box re-reconciles the whole list on every keystroke**
  (`search` is plain `useState` at page level, no debounce, no `useDeferredValue`). This is
  the literal "typing feels unresponsive" experience.
- **Toggling one bookmark re-renders all cards** (`savedWords` array identity changes and the
  whole list subscribes to it; no per-card memoization).

**Fix plan:**
1. Wrap the filter inputs in `useDeferredValue` (React 18 is already in place) so keystrokes
   stay responsive while the list catches up.
2. Cap initial render at ~60 cards with an IntersectionObserver "load more" sentinel
   (windowing without a new dependency; full virtualization is overkill for 500 items once
   paging exists).
3. `React.memo` the card component; move the `saved` lookup into the card via a
   per-id selector (`useProgressStore((s) => s.savedWords.includes(id))`).
4. Drop the per-card mount stagger above ~30 items (animate the container once instead).

### B2. `reducedMotion` setting is a no-op (bug + perf)
`useSettingsStore.reducedMotion` is written by the Settings toggle and **read nowhere**. There
is also no `MotionConfig` and no `prefers-reduced-motion` handling in app CSS. Learners who
turned animations off still get every animation, and low-end devices cannot opt out of the
motion cost.

**Fix plan:** wrap the app in framer's `<MotionConfig reducedMotion={setting ? "always" :
"user"}>` (one line honours both the toggle and the OS setting), and gate the jiggle/stagger
loops on it.

### B3. Three stacked `backdrop-blur-xl` layers on fixed elements (mobile scroll jank)
The sticky header, the desktop sidebar and the bottom tab bar all use `backdrop-blur-xl`
(+ `backdrop-saturate-150`), over a `bg-mesh` gradient body. `backdrop-filter` on fixed
elements forces a repaint of the blurred region every scroll frame; on iOS Safari and low-end
Android this is a classic scroll-jank source (the bar already needs the `translateZ(0)` hack
because of it).

**Fix plan:** reduce to `backdrop-blur-md` and raise surface opacity (e.g. `bg-surface/90`) on
mobile, or use a solid background below the `lg` breakpoint. Visual difference is minimal; the
founder can veto after seeing it live.

### B4. `card-hover` uses `transition-all`
Animates every property on hundreds of cards. Change to
`transition-[transform,box-shadow]`. Small, free win.

### B5. Global search recomputes normalisation on every keystroke
`searchAll` runs `normalise()` (two regex passes) over ~1,100 items × several fields per
keystroke with no debounce. Fine on desktop, noticeable on low-end phones.

**Fix plan:** precompute a normalised index once (module-level, lazily built on first search)
and feed the query through `useDeferredValue`. Combines with fix A1-1.

## C. Bugs (correctness)

### C1. Header streak is stale after a missed day
`AppShell` reads the raw persisted `streak`, while the Dashboard and Fortschritt correctly use
`useEffectiveStreak()` (returns 0 once the streak is broken). After missing a day the header
flame shows e.g. "5" while the dashboard shows 0, until the next activity snaps it down.
**Fix:** one-line change to `useEffectiveStreak()` in AppShell.

### C2. A running session ignores scope/length changes
`SessionPlayer` builds its plan once in a `useState` initializer and `Session.tsx` renders it
without a `key`. Navigating from `/session?theme=A` to `/session?theme=B` (Situationen chip,
global search, any in-app link) does NOT remount the player, so the learner keeps theme A's
deck under a theme B URL. Related: the end screen's "Neue Runde" button is a full
`window.location.reload()`, which re-boots the entire app (and re-runs SW checks) just to
reshuffle. **Fix:** key the player by `(theme, minutes, nonce)` in the route wrapper and make
"Neue Runde" bump the nonce.

### C3. Cloud sync can drop the last seconds of progress
`cloudSync` write-through is debounced 1.5 s with no flush on tab close/backgrounding, and
`signOut()` calls `stopCloudSync()` which **clears a pending push without sending it**. The
mobile pattern "finish session, immediately close the PWA" routinely loses the final review(s)
from the cloud row. The max/union merge bounds the damage but reviews regress across devices.
**Fix:** flush on `visibilitychange === "hidden"` (immediate push, not debounced) and push
once before teardown in `signOut`.

### C4. Minor
- `streak: Math.max(local, remote)` in the cloud merge can resurrect a broken streak across
  devices. Acceptable for now; noting it as a known behavior, not fixing blind.
- `speak()` calls `synth.cancel()` immediately before `synth.speak()`; Chrome has a
  long-standing race that occasionally swallows the utterance. If "TTS sometimes silent" is
  reported, wrap the `speak` in a ~50 ms timeout after `cancel`.

## D. Robustness gaps (why regressions keep reaching the founder)

### D1. CI does not build the app on PRs
`validate.yml` runs content lint + the two engine gates but **not `pnpm typecheck` or
`pnpm build`**. A PR with a type error or broken import merges green and only fails inside the
Pages deploy (or worse, ships a runtime-only break). **Fix:** add typecheck + build (and the
bundle budget, D4) to `validate.yml`.

### D2. No linter
There is no ESLint config at all (some files carry `eslint-disable` comments from editors, but
nothing runs). Rules-of-hooks violations, accidental dead code and unused imports are exactly
the "quietly buggy" class. **Fix:** ESLint flat config with `react-hooks` +
`react-refresh` + `typescript-eslint` recommended, wired into `validate.yml`.

### D3. No component/behavior tests
The only tests are the two engine golden-vector scripts. Nothing asserts that the router
renders, stores migrate, or the session composer respects scope. **Fix:** add Vitest +
Testing Library with a small smoke suite (app boots, each route renders, settings-store
migration, `buildSession` invariants incl. the C2 scope regression). Keep it to ~15 fast tests
so the gate stays cheap.

### D4. No bundle budget
Nothing stops the main chunk creeping past 600 kB again. **Fix:** a ~20-line
`scripts/check-bundle-size.mjs` gate (fail if `index-*.js` exceeds a budget, e.g. 350 kB after
the A1 split) in `validate.yml` after the build step.

### D5. No route-level error boundaries
Only the root boundary exists, so any render error in one page blanks the whole app to the
crash screen. react-router's `errorElement` per route gives a per-page fallback with a
"back to Heute" action. Cheap resilience win.

### D6. Progress store persist has no version
`b2beruf.progress.v1` has no zustand `version`/`migrate`. The next `SrsCard` schema change has
no migration hook and would land on old data shapes silently. Add `version: 0` + a pass-through
migrate now, so the hook exists before it is needed.

## What was checked and found healthy
For completeness: lazy route splitting is already correct (all heavy hubs code-split);
`manualChunks` vendor strategy is sound (incl. the documented react-router TDZ guard); the
stale-SW self-heal (`lib/recover.ts`) is well designed with loop protection; SW registration
is CSP-compliant (external `registerSW.js`, no inline script); the FSRS engine and pronounce
matcher pass their golden-vector gates; zustand selector discipline is good in hot components
(`AppShell`, `BottomTabBar`, `SessionPlayer`); `pnpm audit` reports no known vulnerabilities;
the bottom-bar locked mechanics were not touched by this audit and none of the fixes above
require touching them.
